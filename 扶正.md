# Momo Pet 文档扶正修复计划与验证计划
编码规范严格遵守docs/10-development/CodingSpec.md
## 0. 目标与原则

本计划用于把当前前后端实现与 `docs/` 下的产品文档、功能文档、OpenAPI 文档重新对齐。执行方只按本计划改代码和必要文档，改完后由审查方再次检查。

核心目标：

- 以 `docs/01-product/MVP.md`、`docs/01-product/PRD-MVP-Detail.md`、`docs/02-ux/MVP-ScreenFlow.md`、`docs/06-backend/OpenAPI-MVP.md`、`docs/13-admin/Admin-Console-Spec.md` 为对齐依据。
- 对已经明确相悖的功能，优先改实现；对文档内部互相冲突的地方，先扶正文档口径，再改实现。
- 保持 DDD 分层：Interface -> Application -> Domain -> Infrastructure，Domain 不引入 Spring 注解。
- 前端保持 feature 分层与已有 hooks/components/api 模式，不把流程状态散落到页面组件。
- 每一批改动都必须补测试或至少补可执行验证步骤，不能只靠人工点一点。

## 1. 当前确认的偏差清单

### 1.1 首启流程偏差

文档要求：

- 首次打开：上传宠物照片 -> AI 识别并生成 Pet DNA -> 用户确认名称和性格 -> 生成桌宠。
- UX 文档列出 `WelcomeScreen`、`PetCreateScreen`、`PhotoUploadScreen`、`AiAnalysisScreen`、`PetDnaConfirmScreen`、`DeploySuccessScreen`、`DesktopPetWindow`。

当前实现：

- `apps/desktop/src/features/desktop-pet/hooks/use-default-pet.ts` 会自动创建默认 `Momo Pet`。
- `apps/desktop/src/features/desktop-pet/components/DesktopScene.tsx` 中 Pet Studio 是从桌宠主页打开的可选流程。
- 这会导致用户未完成 Pet DNA 也能进入桌宠主页，与 MVP 用户闭环相悖。

### 1.2 聊天长度契约偏差

文档要求：

- 聊天输入不超过 500 字。

当前实现：

- 后端 `ChatWithPetApplicationService.MAX_MESSAGE_LENGTH = 300`。
- 后端 `SendChatMessageRequest` 使用 `@Size(max = 300)`。
- 前端 `ChatPanel`、`CompactChatInput` 使用 `maxLength={300}`。

### 1.3 AI 任务状态偏差

文档要求：

- AI Pet DNA 任务状态包括 `PENDING`、`RUNNING`、`SUCCEEDED`、`FAILED`、`CANCELED`。

当前实现：

- 后端和共享类型有 `TIMEOUT`，没有 `CANCELED`。
- 前端 Pet Studio 关闭时只是本地取消轮询，不会更新后端任务状态。

### 1.4 清理交互偏差

文档要求：

- 没有清理事件时不调用后端。
- 前端提示“现在很干净。”

当前实现：

- 前端清理动作始终发送 `default-clean-event`。
- 后端没有“当前是否存在清理事件”的查询或判断模型。

### 1.5 Admin Console 范围偏差

文档要求：

- Admin 信息架构包含 Dashboard、Users、Pets、Pet DNA、AI Tasks、Assets、Care Events、Chat Logs、Prompt Library、Analytics、System Settings。
- Dashboard 指标包含今日新增用户、今日新建宠物、Pet DNA 成功率、AI 平均耗时、AI 失败任务、今日互动次数。
- Pet Detail 需要 Overview、Pet DNA、State、Assets、Care Events、Chat Logs、AI Tasks。

当前实现：

- Admin 路由只有 Dashboard、Pets、Pet Detail、AI Tasks。
- Dashboard 只有 total/active pets、AI task 状态数量、recent failures/timeouts。
- Pet Detail 只有 Overview、DNA、State。

### 1.6 OpenAPI 契约偏差

文档当前示例：

- `GET /api/ai/tasks/{taskId}` 返回 `taskType: "PET_DNA"`。

当前实现：

- 代码和 Admin 文档使用 `PET_DNA_GENERATION`。

### 1.7 Create Pet description 偏差

文档要求：

- `POST /api/pets` 请求包含 `description`，表示用户对宠物性格或外观的描述。

当前实现：

- `CreatePetRequest` 有 `description` 字段。
- `PetController.createPet` 调用应用服务时丢弃 `description`。
- Pet 聚合与持久化模型没有 description。

### 1.8 文档内部冲突

冲突点：

- `MVP.md` 将“开机启动开关”列为暂不做。
- `MVP-ScreenFlow.md` 将“开机启动开关”列为 MVP SettingsPanel。

建议口径：

- 以 `MVP.md` 为准，本轮不做开机启动开关。
- SettingsPanel MVP 保留：显示/隐藏桌宠、主动提醒开关、减少动画、退出应用。

### 5.2 AI 任务 CANCELED 支持

需要先定方案。推荐方案：后端真实支持取消状态。

改动文件建议：

- `apps/backend/src/main/java/com/company/momo/ai/domain/AiTaskStatus.java`
- `apps/backend/src/main/java/com/company/momo/ai/domain/AiGenerationTask.java`
- `apps/backend/src/main/java/com/company/momo/ai/application/CancelAiTaskApplicationService.java`
- `apps/backend/src/main/java/com/company/momo/ai/interfaces/AiTaskController.java`
- `packages/shared/src/index.ts`
- `apps/desktop/src/features/pet-studio/hooks/use-pet-studio-ai-flow.ts`
- Onboarding 新流程对应 hook
- `docs/06-backend/OpenAPI-MVP.md`

任务：

- 后端枚举新增 `CANCELED`。
- `AiGenerationTask` 增加 `cancel()` 领域方法：
  - 仅 `PENDING`、`RUNNING` 可取消。
  - `SUCCEEDED`、`FAILED`、`TIMEOUT`、`CANCELED` 不能重复取消。
  - 不能取消时抛业务异常，错误码建议 `AI_TASK_NOT_CANCELABLE`。
- 新增接口：
  - 推荐 `POST /api/ai/tasks/{taskId}/cancel`。
  - 返回最新 task 详情或 `{ taskId, status }`。
- 前端关闭 AI 分析流程时调用 cancel 接口，而不是仅本地停止轮询。
- 轮询逻辑识别 `CANCELED`，展示“已取消”，不进入失败降级。
- OpenAPI 补取消接口和 `CANCELED` 状态。

测试：

- 后端单元测试：`cancel_whenPending_shouldBecomeCanceled`。
- 后端接口测试：取消 PENDING/RUNNING 任务返回 CANCELED。
- 后端接口测试：取消已完成任务返回业务错误。
- 前端人工验证：AI 分析中关闭流程后，Admin AI Task 可看到 CANCELED。

验收：

- `rg -n "CANCELED" apps packages docs` 能看到后端枚举、共享类型、OpenAPI、前端处理。

备选方案：

- 如果产品决定不需要后端取消，只要本地取消轮询，则删除 PRD 中 `CANCELED`，明确这是前端本地行为。该方案会弱化后台审计，不推荐。

### 5.3 清理事件契约扶正

推荐实现：前端根据当前清洁度判断是否有清理事件。MVP 阶段先不引入复杂事件表。

改动文件建议：

- `apps/desktop/src/features/desktop-pet/hooks/use-default-pet.ts`
- `apps/desktop/src/features/desktop-pet/components/PetInteractionLayer.tsx`
- `apps/desktop/src/features/desktop-pet/runtime/default-pet-visuals.ts`
- 如需要，后端 `PetState` 保持不变。

任务：

- 定义 MVP 清理事件规则：
  - 当 `state.cleanliness >= 95` 或 `state.cleanliness >= 100` 时认为没有清理事件。
  - 点击清理不调用后端，提示“现在很干净。”
  - 当 `state.cleanliness < 95` 时才调用后端 `/care/clean`。
- 该阈值抽成常量，避免魔法值。
- 如果产品希望“清理事件”不是 cleanliness 阈值，而是独立事件，则改为后端增加 `GET /api/pets/{petId}/care/clean-events/current`，但这会扩大范围，不建议本轮做。

测试：

- 前端 hook 层如果有测试，补 `cleanliness >= 95` 不调用 API。
- 后端已有 clean 行为测试保留。
- 人工验证：
  - 清洁值满时点清理，只显示提示，不触发 network clean。
  - 清洁值低时点清理，调用后端并增加 cleanliness。

验收：

- 满清洁状态清理不会产生 care/clean 请求。
- 文案精确为“现在很干净。”。

### 5.4 Create Pet description 处理

推荐方案：description 不进入 Pet 聚合，只作为 AI 生成 Pet DNA 的 userDescription；同时扶正文档。

理由：

- 当前 Pet 聚合没有 description，不宜为可选提示词扩展聚合字段。
- Pet DNA 生成已有 `userDescription`，更符合“外观/性格描述用于生成 DNA”的业务含义。

改动文件：

- `docs/01-product/PRD-MVP-Detail.md`
- `docs/06-backend/OpenAPI-MVP.md`
- 前端 Onboarding/Pet Studio 说明文案

任务：

- 将 Create Pet 的 `description` 从 `POST /api/pets` 文档中移除，或标注“仅前端草稿字段，不随 Create Pet 持久化”。
- 在 `POST /api/pets/{petId}/dna/generation-tasks` 文档中明确 `userDescription`。
- 前端创建 pet 时不要发送 description 到 `/api/pets`，而是在创建 AI task 时发送。
- 后端 `CreatePetRequest.description` 建议删除，避免 API 误导；如果兼容需要保留，则标 `@Deprecated` 不推荐，因为接口层 record 不适合长期留无用字段。

测试：

- 更新 Create Pet 接口测试，确认不依赖 description。
- 更新 Pet DNA task 创建测试，确认 `userDescription` 被 AI gateway request 使用。

验收：

- `POST /api/pets` 契约只包含 name/species/birthday。
- description 的唯一用途是 Pet DNA 生成上下文。

### 8.8 回归风险专项检查

重点检查：

- Dashboard 并行查询失败时要有错误响应，不吞异常。
- 删除 `description` 或改口径时，不能破坏 AI 生成 prompt 的 userDescription。

## 10. 推荐执行顺序给其他 AI

1. 先做批次 A，提交文档口径与常量小修。
2. 再做批次 C 的聊天长度，因为风险低、验证明确。
3. 再做批次 B 首启流程，这是最大改动，必须单独验收。
4. 再做批次 C 的 AI cancel 与清理事件。
5. 再做批次 D Admin 文档拆分和指标补齐。
6. 最后做批次 E 文档同步与全量回归。

不要把 B、C、D 混在一个巨大提交里；首启流程、AI cancel、Admin 指标是三个不同风险面，应该分开实现和审查。
