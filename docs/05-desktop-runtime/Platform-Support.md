# Platform Support

## 目标

Project Momo MVP 的桌宠客户端必须同时支持 Windows 和 macOS。

目标不是在两个平台一次性做完所有系统级能力，而是确保同一套产品闭环在两个平台都能使用：

```text
启动应用 -> 显示桌宠 -> 拖动位置 -> 隐藏/恢复 -> 基础互动 -> 重启后恢复
```

## 技术策略

- 使用 Tauri 2 + React + TypeScript 作为统一桌面端技术栈。
- React 负责宠物 UI、互动面板、状态展示和 Pet Studio。
- Rust/Tauri 负责窗口、托盘或菜单栏、文件系统、本地缓存等平台能力。
- 平台差异封装在 desktop runtime adapter 中，前端业务组件不直接判断操作系统。

## MVP 平台矩阵

| 能力 | Windows | macOS | MVP 要求 |
| --- | --- | --- | --- |
| 桌宠窗口启动 | 必须 | 必须 | P0 |
| 透明无边框 | 必须 | 必须 | P0 |
| 基础置顶 | 必须 | 必须 | P0 |
| 拖动宠物 | 必须 | 必须 | P0 |
| 位置保存和恢复 | 必须 | 必须 | P0 |
| 显示/隐藏 | 必须 | 必须 | P0 |
| 托盘或菜单栏退出 | 必须 | 必须 | P0 |
| 鼠标穿透 | 可选 | 可选 | P1 |
| 开机启动 | 不做 | 不做 | 非 MVP |
| 全屏应用自动隐藏 | 不做 | 不做 | 非 MVP |

## 验收标准

- Windows 可以启动桌宠窗口，并通过托盘显示、隐藏和退出。
- macOS 可以启动桌宠窗口，并通过菜单栏入口显示、隐藏和退出。
- 两个平台都能拖动宠物并在重启后恢复位置。
- 两个平台资源缺失时都能回退到 Momo Pet。
- 任一平台窗口能力失败时，应用仍能以普通窗口启动并展示友好提示。

## 风险

- macOS 的 Space、全屏应用和窗口层级行为与 Windows 不同，基础置顶先满足普通桌面场景。
- macOS 菜单栏、签名、公证和权限流程会增加打包验收成本。
- Windows 和 macOS 的多显示器坐标处理不同，位置恢复必须做可见区域校正。
