# AI Developer Working Agreement

## 目标

定义“一名 AI 开发执行 + 一名人工验收”的工作规则，避免 AI 开发时扩大范围、跳过验收或破坏文档约束。

## 工作循环

每个任务按以下流程执行：

```text
读取相关文档
↓
确认本次目标
↓
实现最小可验收功能
↓
自测
↓
汇报变更
↓
人工验收
↓
进入下一任务
```

## 文档优先级

当文档之间出现冲突时，优先级如下：

1. `00-project` 与 `01-product` 产品定义。
2. MVP 验收用例。
3. UI/UX 与 Figma 设计。
4. 后端、桌面端、AI 技术设计。
5. 开发计划与进度表。

开发计划和进度表不能反向扩大产品范围。

## AI 开发执行规则

AI 每次开始编码前必须：

1. 说明本次要做的 Sprint 和任务。
2. 列出会读取或遵循的文档。
3. 确认不做的范围。

AI 编码时必须：

- 遵循 `AI-Coding-Guardrails.md`。
- 不擅自引入新技术栈。
- 不擅自扩大 MVP。
- 不跳过测试或自检。
- 不改无关文件。
- 不重构与任务无关的代码。

AI 完成后必须汇报：

- 改了哪些文件。
- 实现了什么。
- 如何运行。
- 如何验收。
- 有哪些风险。
- 哪些测试已执行。

## 人工验收规则

人工验收每次只验收当前任务。

验收依据：

- `MVP-Acceptance-Cases.md`
- 当前 Sprint 的验收标准
- Figma 当前版本
- README 运行说明

验收结果：

- Pass：进入下一任务。
- Fail：指出具体问题，AI 修复。
- Defer：不影响主流程，记录为后续任务。

## 范围控制

以下需求默认不进入 MVP：

- 宠物交友。
- 商城。
- 支付。
- 插件 SDK。
- 多端同步。
- 复杂 AI Sprite 生成。
- 语音。
- Pet World。

如果确实要加入，必须先更新：

- PRD
- SprintPlan
- Progress Tracker
- Acceptance Cases

## Figma 变更规则

Figma 可以继续精修，但进入开发后：

- 不影响当前 Sprint 的 UI 变更不立刻插队。
- 影响当前功能的变更必须同步到文档。
- 视觉微调可以在 Sprint 8 统一处理。

## API 变更规则

API 一旦实现后，变更必须同步：

- `OpenAPI-MVP.md`
- 前端 API 类型
- 后端测试
- 验收用例

禁止只改代码不改文档。

## Bug 处理规则

Bug 分级：

| 级别 | 说明 | 处理 |
| --- | --- | --- |
| P0 | 阻断主流程 | 立即修 |
| P1 | 影响核心体验但有绕行 | 当前 Sprint 修 |
| P2 | 体验问题 | 记录，Beta 前修 |
| P3 | 优化建议 | 后续处理 |

## 每次任务完成自检

AI 必须确认：

- 是否符合当前 Sprint 目标。
- 是否没有引入非 MVP 功能。
- 是否有运行说明。
- 是否有测试或手动验证步骤。
- 是否没有未完成标记、`console.log`、`printStackTrace`。
- 是否没有敏感日志。
- 是否没有循环内数据库查询或远程调用。
- 是否没有过大的类或方法。

## 进度更新规则

完成任务后更新：

- `MVP-Progress-Tracker.md`

状态只允许使用：

- Not Started
- In Progress
- Review
- Blocked
- Done

人工验收通过后才能标记 Done。
